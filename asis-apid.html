A SIGN IN SPACE — decoding every APID

Field-level reverse engineering of the full ESA/ExoMars-TGO telemetry session that carried the message. The hidden picture is only in APID 0x17; here every one of the 26 streams is decoded from the spp.csv packet log — attitude, thermal, power, command history — with each claim recomputed and adversarially re-verified.

    4b · going deeper — the short/long dump order

    Answering an open question: is the 4B/8B order a hidden channel?

    The community flagged the order of short (4-byte) vs long (8-byte) memory-dump reads as unexplained. Each of the 1800 dumps is one bar below (short / long), in address order. Testing the bytes each read returned settles it: the 8-byte reads land on richer memory — 1.26× denser per byte (), consistent with local data density but only weakly (correlation r≈0.17) — not a proven second channel. Entropy b (message zlib ).
    4c · reconstructing the dumped memory (ID 0x06)

    Is there anything beyond the 8212-byte message?

    The 1800 dumps read memory ID 0x06 from base address 0x0000. The map is the full dumped region: the framed message sits at a clean 0x0000–0x2014 with the header · image body · footer. Bar height = local data density; cyan tick = an 8-byte read. Nothing lies beyond 8212 B — no gaps, no overlaps, no safety-margin bytes; the region was dumped exactly.

    base 0x0000

    Message at logical address zero, region 0x0000–0x2014 (8212 B), one contiguous sweep in strict address order.

    32-bit memory

    Every read is 4-byte aligned; 8-byte reads are two 32-bit words (NOT 8-aligned) ⇒ a 32-bit word-addressable memory, not a 64-bit bus.

    ISO-16 checksum ✓

    The 2-byte trailer is the ECSS ISO-16 checksum over the data bytes only — validates 100% of 1800 dumps (community had ~95%).

    content-driven

    8-byte reads cluster on denser memory and ignore the image’s 32-byte row grid (56/253 straddle a row) — density-aware, 2D-blind.
